Sí, lo ideal sería manejar el Excel con **cuatro columnas fijas**:

```txt
CODIGO_ACTUAL
DESCRIPCION_ACTUAL
CODIGO_NUEVO
DESCRIPCION_NUEVA
```

La lógica correcta sería:

```txt
Buscar CODIGO_ACTUAL en Oracle
Comparar DESCRIPCION_ACTUAL contra la descripción real de Oracle
Comparar DESCRIPCION_ACTUAL contra DESCRIPCION_NUEVA
Mostrar vista previa de cambios
Permitir aprobar o desmarcar filas
Aplicar UPDATE o INSERT solo después de confirmar
```

---

# 1. Impacto y riesgos

## Riesgo principal

El sistema podría reemplazar una descripción o código equivocado si solo se guía por similitud.

Por eso el algoritmo **no debe aplicar cambios automáticamente**. Debe servir para clasificar:

| Similitud  | Estado sugerido            |
| ---------- | -------------------------- |
| 97 - 100   | Alta confianza             |
| 88 - 96    | Revisar                    |
| Menor a 88 | No aplicar automáticamente |

# 3. Regla de negocio corregida

Para cada fila del Excel:

## Caso 1: existe `CODIGO_ACTUAL`

Si el código actual ya existe en `SIS.ITEMS_ISSFA_DETALLE`, se debe revisar si corresponde actualizarlo.

```sql
SELECT
    ID_ITISF,
    CODIGO,
    DESCRIPCION,
    TIPO
FROM SIS.ITEMS_ISSFA_DETALLE
WHERE ID_ITISF = :id_itisf
  AND CODIGO = :codigo_actual;
```

Luego se compara:

```txt
Oracle.DESCRIPCION
vs
Excel.DESCRIPCION_ACTUAL
```

y también:

```txt
Excel.DESCRIPCION_ACTUAL
vs
Excel.DESCRIPCION_NUEVA
```

Si el usuario confirma, se hace:

```sql
UPDATE SIS.ITEMS_ISSFA_DETALLE
SET
    CODIGO = :codigo_nuevo,
    DESCRIPCION = :descripcion_nueva
WHERE ID_ITISF = :id_itisf
  AND CODIGO = :codigo_actual;
```

---

## Caso 2: no existe `CODIGO_ACTUAL`, pero existe `CODIGO_NUEVO`

No se debe insertar duplicado.

Debe mostrarse:

```txt
CODIGO_NUEVO_YA_EXISTE
```

y el usuario debe revisar manualmente.

---

## Caso 3: no existe `CODIGO_ACTUAL` ni `CODIGO_NUEVO`

Se puede proponer insertar:

```sql
INSERT INTO SIS.ITEMS_ISSFA_DETALLE
    (
        ID_ITISF,
        CODIGO,
        DESCRIPCION,
        TIPO
    )
VALUES
    (
        :id_itisf,
        :codigo_nuevo,
        :descripcion_nueva,
        :tipo
    );
```

Pero aquí hay un punto importante: **el campo `TIPO` no viene en el Excel**.

Entonces hay dos opciones seguras:

### Opción recomendada

Agregar una quinta columna:

```txt
TIPO
```

Quedaría así:

```txt
CODIGO_ACTUAL
DESCRIPCION_ACTUAL
CODIGO_NUEVO
DESCRIPCION_NUEVA
TIPO
```

### Opción alternativa

Usar un tipo por defecto seleccionado en pantalla, por ejemplo:

```txt
MED
```

Pero esto es más riesgoso si el Excel mezcla medicamentos, insumos, procedimientos u otros tipos.

# 5. Código base que debe cambiarse

En el código anterior, esta función:

```python
normalize_excel_columns()
```

debe reemplazarse por esta versión:

```python
def normalize_excel_columns(df: pd.DataFrame) -> pd.DataFrame:
    """
    Normaliza nombres de columnas del Excel.

    Formato ideal:
    CODIGO_ACTUAL
    DESCRIPCION_ACTUAL
    CODIGO_NUEVO
    DESCRIPCION_NUEVA

    Opcional:
    TIPO
    """

    aliases = {
        "CODIGO_ACTUAL": [
            "CODIGO_ACTUAL",
            "COD_ACTUAL",
            "CODIGO_ANTERIOR",
            "CODIGO_ORIGINAL",
            "CODIGO_VIEJO",
        ],
        "DESCRIPCION_ACTUAL": [
            "DESCRIPCION_ACTUAL",
            "DESC_ACTUAL",
            "DESCRIPCION_ANTERIOR",
            "DESCRIPCION_ORIGINAL",
            "DESCRIPCION_VIEJA",
        ],
        "CODIGO_NUEVO": [
            "CODIGO_NUEVO",
            "COD_NUEVO",
            "CODIGO_ISSFA",
            "CODIGO_FINAL",
        ],
        "DESCRIPCION_NUEVA": [
            "DESCRIPCION_NUEVA",
            "DESC_NUEVA",
            "DESCRIPCION_ISSFA",
            "DESCRIPCION_FINAL",
        ],
        "TIPO": [
            "TIPO",
            "TIPO_ITEM",
            "CLASE",
        ],
    }

    upper_map = {str(col).strip().upper(): col for col in df.columns}
    renamed = {}

    for canonical, possible_names in aliases.items():
        for name in possible_names:
            if name in upper_map:
                renamed[upper_map[name]] = canonical
                break

    df = df.rename(columns=renamed)

    required = [
        "CODIGO_ACTUAL",
        "DESCRIPCION_ACTUAL",
        "CODIGO_NUEVO",
        "DESCRIPCION_NUEVA",
    ]

    missing = [col for col in required if col not in df.columns]

    if missing:
        raise RuntimeError(
            "El Excel no tiene las columnas requeridas. "
            f"Faltan: {', '.join(missing)}. "
            "Debe tener CODIGO_ACTUAL, DESCRIPCION_ACTUAL, "
            "CODIGO_NUEVO y DESCRIPCION_NUEVA."
        )

    if "TIPO" not in df.columns:
        df["TIPO"] = ""

    return df
```

# 6. Nueva lógica de análisis

También se debe cambiar la función de análisis para que ya no piense en “código anterior contra código ISSFA”, sino en:

```txt
actual → nuevo
```

Código recomendado:

```python
def analyze_excel(conn, excel_path: str, id_itisf: int, threshold: float) -> list[dict]:
    """
    Lee Excel y genera vista previa.

    No inserta ni actualiza nada.
    Solo determina qué acción se podría realizar:
    - UPDATE
    - INSERT
    - BLOQUEADO
    - REVISAR
    """

    df = pd.read_excel(excel_path)
    df = normalize_excel_columns(df)

    result: list[dict] = []

    for index, row in df.iterrows():
        codigo_actual = str(row["CODIGO_ACTUAL"]).strip()
        descripcion_actual_excel = str(row["DESCRIPCION_ACTUAL"]).strip()
        codigo_nuevo = str(row["CODIGO_NUEVO"]).strip()
        descripcion_nueva = str(row["DESCRIPCION_NUEVA"]).strip()
        tipo_excel = str(row.get("TIPO", "")).strip()

        if not codigo_nuevo or codigo_nuevo.lower() == "nan":
            continue

        if not descripcion_nueva or descripcion_nueva.lower() == "nan":
            continue

        actual_db = find_detalle_by_codigo(
            conn=conn,
            id_itisf=id_itisf,
            codigo_issfa=codigo_actual,
        )

        nuevo_db = find_detalle_by_codigo(
            conn=conn,
            id_itisf=id_itisf,
            codigo_issfa=codigo_nuevo,
        )

        score_oracle_vs_excel = 0.0
        score_actual_vs_nueva = similarity_score(
            descripcion_actual_excel,
            descripcion_nueva,
        )

        descripcion_actual_oracle = ""
        tipo_final = tipo_excel

        if actual_db:
            descripcion_actual_oracle = actual_db.get("DESCRIPCION") or ""
            tipo_final = actual_db.get("TIPO") or tipo_excel

            score_oracle_vs_excel = similarity_score(
                descripcion_actual_oracle,
                descripcion_actual_excel,
            )

        accion = "BLOQUEADO"
        status = ""
        aplicar = False

        if actual_db and not nuevo_db:
            accion = "UPDATE"
            status = "ACTUALIZAR_CODIGO_Y_DESCRIPCION"

            aplicar = (
                score_oracle_vs_excel >= threshold
                and score_actual_vs_nueva >= threshold
            )

        elif actual_db and nuevo_db and codigo_actual != codigo_nuevo:
            accion = "BLOQUEADO"
            status = "CODIGO_NUEVO_YA_EXISTE"
            aplicar = False

        elif actual_db and nuevo_db and codigo_actual == codigo_nuevo:
            accion = "UPDATE"
            status = "ACTUALIZAR_SOLO_DESCRIPCION"
            aplicar = score_oracle_vs_excel >= threshold

        elif not actual_db and not nuevo_db:
            accion = "INSERT"
            status = "INSERTAR_NUEVO"

            aplicar = bool(tipo_final) and score_actual_vs_nueva >= threshold

            if not tipo_final:
                status = "FALTA_TIPO_PARA_INSERTAR"
                aplicar = False

        elif not actual_db and nuevo_db:
            accion = "BLOQUEADO"
            status = "CODIGO_NUEVO_YA_EXISTE_SIN_ACTUAL"
            aplicar = False

        decision = decision_from_score(
            max(score_oracle_vs_excel, score_actual_vs_nueva),
            threshold,
        )

        result.append(
            {
                "APLICAR": aplicar,
                "FILA_EXCEL": index + 2,
                "ACCION": accion,
                "STATUS": status,
                "DECISION": decision,
                "ID_ITISF": id_itisf,
                "CODIGO_ACTUAL": codigo_actual,
                "DESCRIPCION_ACTUAL_EXCEL": descripcion_actual_excel,
                "DESCRIPCION_ACTUAL_ORACLE": descripcion_actual_oracle,
                "CODIGO_NUEVO": codigo_nuevo,
                "DESCRIPCION_NUEVA": descripcion_nueva,
                "TIPO": tipo_final,
                "SCORE_ORACLE_EXCEL": score_oracle_vs_excel,
                "SCORE_ACTUAL_NUEVA": score_actual_vs_nueva,
                "EXISTE_ACTUAL": bool(actual_db),
                "EXISTE_NUEVO": bool(nuevo_db),
            }
        )

    return result
```

# 7. Nueva función para aplicar cambios

Se debe crear esta función:

```python
def update_detalle_codigo_descripcion(
    conn,
    id_itisf: int,
    codigo_actual: str,
    codigo_nuevo: str,
    descripcion_nueva: str,
) -> int:
    """
    Actualiza código y descripción en ITEMS_ISSFA_DETALLE.
    """

    sql = """
        UPDATE SIS.ITEMS_ISSFA_DETALLE
        SET
            CODIGO = ?,
            DESCRIPCION = ?
        WHERE ID_ITISF = ?
          AND CODIGO = ?
    """

    return execute_non_query(
        conn,
        sql,
        (
            codigo_nuevo,
            descripcion_nueva,
            id_itisf,
            codigo_actual,
        ),
    )
```

Y la función de aplicar debe trabajar así:

```python
def apply_changes(conn, rows: list[dict], audit_path: str) -> None:
    """
    Aplica cambios en una sola transacción.
    Si algo falla, se hace rollback.
    """

    applied_rows = []

    try:
        for row in rows:
            if not row.get("APLICAR"):
                continue

            accion = row["ACCION"]

            if accion == "UPDATE":
                affected = update_detalle_codigo_descripcion(
                    conn=conn,
                    id_itisf=int(row["ID_ITISF"]),
                    codigo_actual=row["CODIGO_ACTUAL"],
                    codigo_nuevo=row["CODIGO_NUEVO"],
                    descripcion_nueva=row["DESCRIPCION_NUEVA"],
                )

                audit = dict(row)
                audit["ROWS_AFFECTED"] = affected
                audit["APPLIED_AT"] = datetime.now().isoformat(timespec="seconds")
                applied_rows.append(audit)

            elif accion == "INSERT":
                inserted = insert_detalle_if_missing(
                    conn=conn,
                    id_itisf=int(row["ID_ITISF"]),
                    codigo_issfa=row["CODIGO_NUEVO"],
                    descripcion_issfa=row["DESCRIPCION_NUEVA"],
                    tipo=row["TIPO"],
                )

                audit = dict(row)
                audit["INSERTED_DETALLE"] = inserted
                audit["APPLIED_AT"] = datetime.now().isoformat(timespec="seconds")
                applied_rows.append(audit)

            else:
                continue

        conn.commit()

    except Exception:
        conn.rollback()
        raise

    write_audit_csv(audit_path, applied_rows)
```

# 8. Advertencia importante sobre equivalencias

Si se actualiza el código en `SIS.ITEMS_ISSFA_DETALLE`, también puede existir dependencia en:

```txt
SIS.EQUIVALENCIAS_ITEMS_ISSFA.CODIGO_ISSFA
```

Por eso hay que validar antes:

```sql
SELECT COUNT(*)
FROM SIS.EQUIVALENCIAS_ITEMS_ISSFA
WHERE ID_ITISF = :id_itisf
  AND CODIGO_ISSFA = :codigo_actual;
```

Si existe, hay dos alternativas:

## Alternativa segura

Bloquear el cambio de código y solo permitir cambiar descripción.

```txt
BLOQUEADO_TIENE_EQUIVALENCIAS
```

## Alternativa completa

Actualizar también la equivalencia:

```sql
UPDATE SIS.EQUIVALENCIAS_ITEMS_ISSFA
SET CODIGO_ISSFA = :codigo_nuevo
WHERE ID_ITISF = :id_itisf
  AND CODIGO_ISSFA = :codigo_actual;
```

La opción más segura para producción es:

```txt
Si tiene equivalencias, bloquear cambio de código.
Solo permitir cambio de descripción.
```

# 9. Instrucción clara para el programador

El programador debe cambiar la lógica para que el Excel tenga cuatro columnas obligatorias:

```txt
CODIGO_ACTUAL
DESCRIPCION_ACTUAL
CODIGO_NUEVO
DESCRIPCION_NUEVA
```

La aplicación debe buscar `CODIGO_ACTUAL` en `SIS.ITEMS_ISSFA_DETALLE`. Si existe, debe mostrar una vista previa indicando que se hará `UPDATE`. Si no existe, debe revisar si `CODIGO_NUEVO` ya existe. Si tampoco existe, puede proponer `INSERT`, siempre que exista un `TIPO` válido.

La aplicación debe comparar descripciones con Jaro-Winkler y Levenshtein, pero la similitud solo debe ayudar a decidir visualmente. No debe aplicar cambios sin que el usuario vea la vista previa y confirme.

No debe tocar `ITEMS_ISSFA_CABECERA`.

No debe actualizar registros que tengan equivalencias asociadas sin validación previa.
